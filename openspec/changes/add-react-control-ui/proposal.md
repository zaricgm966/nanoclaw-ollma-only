# 提案：增加 React 控制台

## 背景

NanoClaw 当前主要通过聊天渠道、SQLite 状态、日志文件以及直接改代码来运作。
这种方式保持了运行时足够轻量，但在日常使用里，监控运行状态、查看会话、排查问题的成本仍然偏高。
一个本地 Web 控制台可以在不改变 NanoClaw 消息驱动本质的前提下，显著降低这些操作的门槛。

同时，项目也需要一套可持续的变更记录机制。此次改动会一并引入 OpenSpec 工作流，为后续 UI、平台和渠道扩展建立规范化记录方式。

## 本次改动内容

本提案引入：

- 基于 React 的本地 Web 控制台
- 由现有 NanoClaw Node 进程暴露的轻量 HTTP API 层
- 用于状态和日志更新的实时或准实时能力
- 用于记录后续重大改动的 OpenSpec 工作流

本次 change 刻意把范围限制在基础层：

- UI 应用骨架
- API 骨架
- 仪表盘/状态页
- 群组和日志可见性
- OpenSpec 项目结构

本次不要求：

- 完整的 UI 写操作能力
- 浏览器内复杂鉴权流程
- 多用户权限系统
- 一次性覆盖未来所有 NanoClaw 渠道能力

## 目标

- 提供一个接近 OpenClaw 风格的控制台体验，同时保持 NanoClaw 的轻量架构
- 保持 NanoClaw 仍然以单个主 Node 进程为核心，而不是拆成多个后端服务
- 明显提升本地调试和状态查看体验
- 建立 OpenSpec 作为后续重大改动的标准流程

## 非目标

- 把 NanoClaw 改造成云端 SaaS
- 把聊天优先模式改成 UI 优先模式
- 引入复杂微服务后端
- 在第一阶段就交付完整的管理写操作

## 成功标准

- 仓库内具备可持续使用的 OpenSpec 结构
- React 应用可以在当前仓库内独立开发，而不破坏既有运行流程
- NanoClaw 暴露出面向仪表盘、群组、会话与日志的最小 API 面
- 第一版 UI 可以展示来自现有系统的实时或近期数据
